The Solana dApp Store accepts **signed APK** files. This guide covers how to build and sign an APK for your project, regardless of framework.

## Create a signing key

Every app needs its own signing key. For a full understanding of how Android app signing works, see the [official Android documentation](https://developer.android.com/studio/publish/app-signing#certificates-keystores).

Generate a keystore for your app:

```bash
keytool -genkey -v -keystore my-app-name.keystore \
  -alias my-app-name \
  -keyalg RSA \
  -keysize 2048 \
  -validity 10000
```

Replace `my-app-name` with your app's name.

Store the keystore file and passwords securely — losing them means you cannot update your app. All future updates must be signed with the same key.

<Warning>
If your app is also published on Google Play, you **must** use a separate signing key for the dApp Store. You cannot reuse the same key for both stores.
</Warning>

<Accordion title="Publishing multiple apps?">
Each app should have its own signing key. You can keep things organized by storing multiple key aliases in a single keystore file:

```bash
# First app
keytool -genkey -v -keystore publisher.keystore -alias my-first-app -keyalg RSA -keysize 2048 -validity 10000

# Second app
keytool -genkey -v -keystore publisher.keystore -alias my-second-app -keyalg RSA -keysize 2048 -validity 10000
```

Then reference the appropriate `-alias` when signing each app.
</Accordion>

By default, EAS builds an Android App Bundle (.aab). The dApp Store requires an APK instead.

For signing, EAS can automatically generate and manage a keystore for you on your first build. Alternatively, you can provide your own keystore created with the [instructions above](#create-a-signing-key).

## 1. Configure EAS for APK builds

Add a `dapp-store` build profile to your `eas.json`:

```json eas.json
{
  "build": {
    "dapp-store": {
      "android": {
        "buildType": "apk"
      }
    }
  }
}
```

## 2. Build the APK

<Tabs>
<Tab title="EAS Cloud">

```bash
eas build --platform android --profile dapp-store
```

The build runs on EAS servers. Once finished, download the APK from the link provided.

</Tab>
<Tab title="Local Build">

```bash
eas build --platform android --profile dapp-store --local
```

The build runs on your machine. Requires the [Android SDK and NDK](https://docs.expo.dev/get-started/set-up-your-environment/?mode=development-build&buildEnv=local) installed locally. The APK is output to the project directory.

</Tab>
</Tabs>

On your first build, EAS will prompt you to either generate a new keystore automatically or provide an existing one.

## Verify the APK

Confirm your APK is properly signed:

```bash
apksigner verify --print-certs app-release.apk
```

## 1. Configure signing in build.gradle

Add a signing config in your `app/build.gradle`:

```kotlin app/build.gradle
android {
    signingConfigs {
        dappStore {
            storeFile file("keystores/my-app-name.keystore")
            storePassword "your_keystore_password"
            keyAlias "my-app-name"
            keyPassword "your_key_password"
        }
    }

    buildTypes {
        release {
            signingConfig signingConfigs.dappStore
        }
    }
}
```

## 2. Build the APK

```bash
./gradlew assembleRelease
```

Or in Android Studio: **Build** > **Build APK(s)**.

Your signed APK will be at `app/build/outputs/apk/release/app-release.apk`.

## 3. Verify the APK

Confirm your APK is properly signed:

```bash
apksigner verify --print-certs app-release.apk
```

<Tab title="Web Shell" icon="globe">

The [Solana Mobile Web Shell CLI](https://github.com/solana-mobile/web-shell/) helps developers generate an Android app project that wraps an existing web app or website inside an Android WebView.

<Note>
**Why Web Shell instead of Bubblewrap?** Browsers are rolling out [Local Network Access](https://developer.chrome.com/blog/local-network-access) restrictions that can break wallet association flows on Android web, especially flows that rely on local network or loopback communication. Android web apps and native app wrappers that use Custom Chrome Tabs, including Bubblewrap-style APKs, can be affected by this change. Web Shell exists to work around those browser-side restrictions by moving the app into an Android WebView while preserving Solana Mobile wallet flows. Because the site is loaded directly inside the app, Web Shell does **not** require Digital Asset Links.
</Note>

### Prerequisites

- Node.js **24 or newer** (`node -v`)
- A web app or website reachable at a public URL
- The web app must use `@solana-mobile/wallet-adapter-mobile@^2.2.8` or newer — older releases will silently fail inside Web Shell. See [Wallet adapter compatibility](#wallet-adapter-compatibility) below.

<Accordion title="Template: Web Manifest">
At minimum, a manifest file includes the app's name, icons, and start URL:

```json manifest.json
{
    "name": "APP_NAME",
    "short_name": "APP_NAME",
    "scope": "/",
    "start_url": "/",
    "icons": [
        {
            "src": "/android-chrome-192x192.png",
            "sizes": "192x192",
            "type": "image/png"
        },
        {
            "src": "/android-chrome-512x512.png",
            "sizes": "512x512",
            "type": "image/png"
        }
    ],
    "theme_color": "#ffffff",
    "background_color": "#ffffff",
    "display": "standalone"
}
```

For more information, view the [web manifest documentation](https://web.dev/articles/add-manifest).
</Accordion>

## 1. Install the Web Shell CLI

```bash
npm install -g @solana-mobile/webshell-cli
```

## 2. Initialize the project

In a new directory, run `init`. If your site is a [Progressive Web App](https://developer.mozilla.org/en-US/docs/Web/Progressive_web_apps) with a [web manifest](https://developer.mozilla.org/en-US/docs/Web/Manifest), point at the manifest so Web Shell can auto-fill the app name, icons, theme, and start URL from it:

```bash
webshell init ./my-app --manifest https://your-app.com/manifest.json
```

For a plain web app or website without a manifest, point at the site URL instead:

```bash
webshell init ./my-app --url https://your-app.com
```

Either form prompts you to configure any remaining settings — application ID, app name, version code/name, and signing keystore.

<Warning>
`init` prompts for a keystore path and alias and creates a new **Android Keystore** at that path if one does not already exist (existing keystores are reused). Keep the keystore file and passwords secure — losing them means you cannot update your app.
</Warning>

## 3. Build the APK

```bash
webshell build ./my-app
```

The first build may take longer than subsequent ones — Web Shell automatically installs any missing toolchain pieces (JDK 17, Android SDK, `platform-tools`, `build-tools`).

You'll be prompted for the keystore password and, if different, the signing key password. To skip the prompts in CI, set them as environment variables:

```bash
export WEB_SHELL_KEYSTORE_PASSWORD='<keystore-password>'
export WEB_SHELL_KEY_PASSWORD='<key-password>'
webshell build ./my-app
```

The signed APK is written to `my-app/app/build/outputs/apk/release/app-release.apk`.

## 4. Test the APK

Install on a connected device or emulator:

```bash
adb install -r my-app/app/build/outputs/apk/release/app-release.apk
```

## Wallet adapter compatibility

Web Shell runs your site inside an Android WebView. For Solana Mobile wallet flows to keep working there, your web app must use a Web Shell–compatible release of the Mobile Wallet Adapter:

```bash
npm install @solana-mobile/wallet-adapter-mobile@^2.2.8
```

<Warning>
This is required, not optional. Older releases will not detect the Web Shell environment and wallet association will silently fail.
</Warning>

### Updating your app

To release a new version, bump `WEB_SHELL_VERSION_CODE` and `WEB_SHELL_VERSION_NAME` in the generated project's `gradle.properties` and re-run `build`:

```bash
webshell build ./my-app
```
